# Add delete-book functionality

> **Status (2026-04-29):** implemented. `tsc --noEmit` clean; not yet exercised in a browser.

## Context

`oh-book-reader` had no way to remove an uploaded PDF — confirmed by searching for any `DELETE` handler, `fs.unlink`/`fs.rm` call, or remove-book UI; none existed. Books accumulated forever in `data/books/`. The user opened the app expecting a delete control on the library and couldn't find one.

Each book is a self-contained directory `data/books/b_<ulid>/` holding `book.pdf`, `meta.json`, plus `selections/` and `conversations/` subdirs. So deletion is a single recursive directory removal — no DB rows, no shared/external state.

## Approach

Three surgical changes, each mirroring an existing pattern in the repo.

### 1. `lib/store.ts` — add `deleteBook`

Added next to `saveBook`:

```ts
export async function deleteBook(bookId: string): Promise<void> {
  await fs.rm(bookDir(bookId), { recursive: true, force: true });
}
```

`fs.rm({ recursive: true, force: true })` wipes the entire `b_<id>/` dir — PDF, meta, selections, conversations — and is a no-op if it's already gone, which is the behavior we want for an idempotent DELETE.

### 2. New file `app/api/books/[id]/route.ts` — DELETE handler

There was no `route.ts` at `app/api/books/[id]/` (only the `file/` and `selections/` subroutes). Created one with a DELETE handler that mirrors the param-handling style of `app/api/books/[id]/file/route.ts` (Next.js 16 — `params` is a Promise, per `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/route.md`):

```ts
import { NextRequest, NextResponse } from "next/server";
import { deleteBook, getBook } from "@/lib/store";

export const runtime = "nodejs";

const BOOK_ID_RE = /^b_[0-9A-HJKMNP-TV-Z]+$/;

export async function DELETE(
  _req: NextRequest,
  ctx: { params: Promise<{ id: string }> },
) {
  const { id } = await ctx.params;
  if (!BOOK_ID_RE.test(id)) {
    return new Response("not found", { status: 404 });
  }
  try {
    await getBook(id);
  } catch {
    return new Response("not found", { status: 404 });
  }
  await deleteBook(id);
  return NextResponse.json({ ok: true });
}
```

The regex enforces the Crockford base32 ULID alphabet (`newBookId()` output) — defense-in-depth against path traversal like `../etc`. The `getBook(id)` pre-check then returns a clean 404 for IDs that match the shape but don't exist on disk.

### 3. `app/page.tsx` — delete button per row

The list item used to wrap the entire row in a single `<Link>`. Restructured to a flex row where `<Link>` covers only the title and a `<button>` triggers delete:

- Delete handler: `window.confirm` → `fetch('/api/books/' + b.id, { method: 'DELETE' })` → on `r.ok`, call the existing `refresh()`; otherwise alert with the response text (matches the existing upload-error pattern).
- Added a `deleting` state (Set<string> of in-flight ids) so the button shows "Deleting…" and is disabled during the request — same shape as `uploading`.

The reader page (`/books/[bookId]`) intentionally does not get a delete button — keeping the action in the library avoids the question of "what should I redirect to after deleting the book I'm reading."

## Verification

1. `npm run dev`.
2. Upload a PDF via the existing upload button, then make at least one selection in the reader so `selections/` is populated.
3. Back on the library, click delete → confirm → row disappears, list refreshes.
4. Verify `data/books/<id>/` is gone on disk: `ls data/books/`.
5. `curl -X DELETE http://localhost:3000/api/books/b_doesnotexist` → 404.
6. `curl -X DELETE http://localhost:3000/api/books/../etc` → 404 (regex rejects).
7. Re-upload a PDF and confirm upload still works post-delete (no stale state).
8. `npx tsc --noEmit` passes (already verified at implementation time).
