# Auto-quote mermaid node labels with risky characters

## Context

In the conversation thread view, mermaid diagrams generated by Claude fail to render
when a node label contains parentheses, e.g. `C[Gradient ∇f(x)]`. Mermaid parses the
`(` inside `[...]` as the start of a different shape and aborts with a parse error
(`Expecting 'SQE'... got 'PS'`). The renderer (`components/MermaidDiagram.tsx:49`)
passes the source string straight to `mermaid.render` with no preprocessing, so any
diagram with parentheses inside a square-bracket label dies.

Fix: preprocess the source before handing it to mermaid by wrapping unquoted node
labels in double quotes when the label content contains characters that would
otherwise be misparsed. Quoted labels are mermaid's official escape hatch — they
accept arbitrary content. The fix is contained to one file and does not change the
error fallback or copy-button source (which keep the original `code` prop).

## Approach

Add a pure helper `quoteRiskyMermaidLabels(code: string): string` at the top of
`components/MermaidDiagram.tsx`, and call it on `code` inside the existing render
effect (line 49) — `await m.render(id, quoteRiskyMermaidLabels(code))`.
The original `code` prop continues to feed the loading/error/copy paths
(lines 67, 79, 93), so users still see what they actually wrote on failure.

The helper runs **three independent regex passes** — one per node-label shape —
each carefully scoped to skip compound shapes that share the opening character.

### Trigger character class

```js
const TRIGGER = /[(){}\[]/;
```

If the captured label body contains none of these, the match is left unchanged
(keeps source clean — only quote when needed).

### Regexes

```js
// Rectangle:  id[text]   (excludes [[, [(, [/, [\, [")
/(^|[\s\->|&;])([A-Za-z0-9_]+)\[(?!["[(/\\])([^\]\n]*?)\](?!\])/g

// Rhombus:    id{text}   (excludes {{, {")
/(^|[\s\->|&;])([A-Za-z0-9_]+)\{(?!["{])([^{}\n]*?)\}(?!\})/g

// Rounded:    id(text)   (excludes ((, (")  — body forbids ( ) so labels
//                          containing parens won't match this pass; that's a
//                          known limitation, see Risks
/(^|[\s\->|&;])([A-Za-z0-9_]+)\((?!["(])([^()\n]*?)\)(?!\))/g
```

The leading `(^|[\s\->|&;])` is the prefix anchor: an id must follow line start,
whitespace (incl. `\n`), or edge syntax (`-->`, `--`, `&`, `;`). The negative
lookahead after the opener excludes compound shapes (`[[`, `[(`, `[/`, `[\`,
`((`, `{{`) and already-quoted labels (`["`, `("`, `{"`). The negative lookahead
on the close prevents matching just the inner half of a doubled close (`]]`,
`}}`, `))`).

### Replacement

For each match where `TRIGGER.test(body)`:

1. Escape any literal `"` in body as `#quot;` (mermaid HTML-entity convention).
2. Re-emit `${prefix}${id}${open}"${escaped}"${close}`.

Idempotent by construction: re-running the helper on its own output is a no-op
because the opener lookahead now sees `["` / `("` / `{"` and skips.

### Sketch

```js
function quoteRiskyMermaidLabels(src: string): string {
  const TRIGGER = /[(){}\[]/;
  const esc = (s: string) => s.replace(/"/g, "#quot;");
  return src
    .replace(/(^|[\s\->|&;])([A-Za-z0-9_]+)\[(?!["[(/\\])([^\]\n]*?)\](?!\])/g,
      (m, p, i, b) => TRIGGER.test(b) ? `${p}${i}["${esc(b)}"]` : m)
    .replace(/(^|[\s\->|&;])([A-Za-z0-9_]+)\{(?!["{])([^{}\n]*?)\}(?!\})/g,
      (m, p, i, b) => TRIGGER.test(b) ? `${p}${i}{"${esc(b)}"}` : m)
    .replace(/(^|[\s\->|&;])([A-Za-z0-9_]+)\((?!["(])([^()\n]*?)\)(?!\))/g,
      (m, p, i, b) => TRIGGER.test(b) ? `${p}${i}("${esc(b)}")` : m);
}
```

## Verification

Manual smoke test in the dev server (`npm run dev`):

1. Reproduce the user's failing diagram in a new conversation thread:

   ```
   graph TD
       A[Function f: R^n → R] --> B[Differentiable?]
       B -->|Yes| C[Gradient ∇f(x)]
       B -->|Twice| D[Hessian ∇²f(x)]
       C --> E[Vector in R^n]
       D --> F[n × n Symmetric Matrix]
       F --> G[Entry i,j: ∂²f/∂xᵢ∂xⱼ]
   ```

   Expect: renders as a flowchart instead of showing the parse-error `<details>`.

2. Confirm previously-working diagrams still render:
   - A simple labeled rectangle: `A[Hello] --> B[World]` (no triggers, unchanged).
   - A cylinder: `DB[(Database)]` (must NOT be re-quoted to `DB[("Database")]`).
   - A subroutine: `S[[Init]]` (must NOT be touched).
   - A parallelogram: `P[/Step/]` (must NOT be touched).
   - A diagram with an already-quoted label: `Q["Already (quoted)"]` (unchanged).

3. Confirm the copy-button output and the error-fallback `<pre>` (lines 67, 79, 93)
   still show the **original** unmodified source — verify by copying from a
   diagram whose label was preprocessed.

4. Re-run the preprocessor on its own output (mentally) for case 1 — must be
   identical (idempotent).

## Risks

- **Rounded labels with parens** (`X(f(x))`) are not auto-quoted because the
  rounded-pass body class `[^()\n]` forbids parens. Acceptable: the user's
  failing case is rectangles, and mermaid users typically reach for rounded
  shapes for short labels. Can add a fourth pass with balanced-paren matching
  later if needed.
- **Diagram-type directives** (`graph TD`, `flowchart LR`, `classDef foo fill:#f9f`,
  `style A fill:#fff,stroke:#000`, `linkStyle …`) contain no `[` / `(` / `{`
  after an id token in their syntax, so the regex won't accidentally rewrite
  them.
- **`subgraph` titles**: `subgraph "Title (x)"` is already quoted (unchanged);
  `subgraph s1 [Title]` has a space between id and `[` (won't match — good);
  `subgraph s1[Title (x)]` would match and be quoted (benign — quoting a
  subgraph title with parens is correct).
- **Performance**: three regex passes over a typically <2 KB string, negligible.
